# HRMS Lean V1 Implementation Plan

## Summary

Build a lean, production-ready HRMS MVP that is sellable quickly and intentionally limited to the core business flows: authentication, employee management, attendance, leave approval, simple payroll, and role-based dashboards. Keep the architecture modular and Prisma-backed so V2 features can be added without rewriting the core schema or API structure.

## Key Changes

### Product and architecture scope
- Keep the stack as `React + Express + Prisma + PostgreSQL + JWT`.
- Use fixed application roles in V1: `ADMIN`, `HR`, `MANAGER`, `EMPLOYEE`.
- Implement role checks in code; do not introduce a permission engine yet.
- Treat V1 as single-company only; no multi-tenant logic.

### Backend implementation
- Create `backend/` with modular folders for `auth`, `employees`, `departments`, `attendance`, `leaves`, `payroll`, and `dashboard`.
- Set up Express app foundation with config loading, request validation, auth middleware, role guard middleware, error middleware, and uniform API response format.
- Implement Prisma schema for:
  - `Role`
  - `User`
  - `Department`
  - `Employee`
  - `Attendance`
  - `LeaveType`
  - `LeaveBalance`
  - `LeaveRequest`
  - `PayrollRecord`
- Add required constraints and indexes:
  - unique `users.email`
  - unique `employees.user_id`
  - unique `employees.employee_code`
  - unique `attendance(employee_id, attendance_date)`
  - unique `leave_balances(employee_id, leave_type_id, year)`
  - unique `payroll_records(employee_id, month, year)`
  - indexes on manager, employee, date, status, and department lookup paths
- Seed fixed roles, one admin user, and baseline leave types.

### Auth and security
- Implement only:
  - `POST /api/auth/login`
  - `POST /api/auth/logout`
  - `GET /api/auth/me`
- Use bcrypt password hashing and JWT access tokens.
- No refresh token system in V1.
- Block inactive users from login and protected APIs.
- Return current user, linked employee, and role from `/api/auth/me`.

### Core modules
- Employees:
  - employee CRUD
  - profile retrieval
  - status update
  - soft delete using `is_active` or `deleted_at`
- Departments:
  - department CRUD
- Attendance:
  - check-in
  - check-out
  - attendance list
  - employee attendance history
  - one record per employee per day
  - reject duplicate check-in and invalid checkout
- Leaves:
  - leave types
  - self leave balances
  - apply leave
  - leave listing
  - approve leave
  - reject leave
  - prevent overlap
  - enforce manager or HR approval
  - deduct leave balance only on approval
- Payroll:
  - create monthly payroll record
  - update draft payroll record
  - list payroll by employee or role scope
  - keep payroll fields simple: salary, month, year, status
- Dashboard:
  - employee dashboard
  - manager dashboard
  - HR/admin dashboard
  - keep metrics simple and live-query based in V1

### Frontend implementation
- Create `frontend/` with feature folders aligned to backend modules.
- Build:
  - login flow
  - protected app shell
  - role-aware navigation
  - dashboard pages
  - employee management pages
  - department management pages
  - attendance pages
  - leave apply/history/approval pages
  - payroll list/history pages
- Use `/api/auth/me` as the source of truth for session restoration and route protection.
- Keep the UI simple, fast, and internal-tool oriented.

### Delivery order
1. Scaffold monorepo structure and package setup.
2. Build backend app shell, Prisma configuration, and database schema.
3. Add seed data for roles, admin user, and leave types.
4. Implement auth, JWT middleware, and role guards.
5. Implement departments and employees.
6. Implement attendance workflows.
7. Implement leave balances and leave approval workflow.
8. Implement simple payroll module.
9. Implement dashboard endpoints.
10. Build frontend screens against stable APIs.
11. Add basic tests, env examples, and local setup documentation.

## Public Interfaces

### API surface for V1
- Auth:
  - `POST /api/auth/login`
  - `POST /api/auth/logout`
  - `GET /api/auth/me`
- Employees:
  - `GET /api/employees`
  - `GET /api/employees/:id`
  - `POST /api/employees`
  - `PUT /api/employees/:id`
  - `PATCH /api/employees/:id/status`
- Departments:
  - `GET /api/departments`
  - `POST /api/departments`
  - `PUT /api/departments/:id`
- Attendance:
  - `POST /api/attendance/check-in`
  - `POST /api/attendance/check-out`
  - `GET /api/attendance`
  - `GET /api/attendance?employeeId=`
- Leaves:
  - `GET /api/leave-types`
  - `GET /api/leave-balances/me`
  - `GET /api/leaves`
  - `POST /api/leaves`
  - `PUT /api/leaves/:id/approve`
  - `PUT /api/leaves/:id/reject`
- Payroll:
  - `GET /api/payroll`
  - `POST /api/payroll`
  - `PUT /api/payroll/:id`
- Dashboard:
  - `GET /api/dashboard/employee`
  - `GET /api/dashboard/manager`
  - `GET /api/dashboard/hr`

### Data model decisions
- `Users.role_id` is the only access-control source in V1.
- `Employees.manager_id` is required for manager approval and team visibility.
- `LeaveRequest` must store `status`, `approved_by`, `approved_at`, `created_at`, `updated_at`.
- `PayrollRecord` is monthly and unique per employee per month and year.
- Sensitive file/document storage is excluded from V1.

## Test Plan

- Auth:
  - valid login succeeds
  - invalid credentials fail
  - inactive user is blocked
  - protected route rejects missing or invalid token
- RBAC:
  - employees cannot access admin or HR routes
  - managers only access team approval flows
  - HR/admin can access operational modules
- Employees:
  - duplicate email rejected
  - duplicate employee code rejected
  - soft-deleted or inactive employee behavior is enforced
- Attendance:
  - duplicate attendance for same date rejected
  - checkout before check-in rejected
  - employee history filter works
- Leaves:
  - overlapping leave rejected
  - insufficient balance rejected
  - self-approval rejected
  - manager approval updates status and balance
  - rejection does not deduct balance
- Payroll:
  - duplicate payroll for same month and year rejected
  - payroll update allowed only before locked business rule if implemented
  - employee sees only permitted payroll records
- Dashboard:
  - role-specific metrics return correct scoped data

## Assumptions

- V1 prioritizes delivery speed over enterprise completeness.
- Role-based access in code is sufficient for the initial commercial release.
- JWT access-token-only auth is acceptable for V1.
- Leave types and leave balances are included in V1 because they are foundational, not overengineering.
- Audit logs, permissions, refresh tokens, documents, notifications, shifts, designations, and advanced reporting are deferred to V2.
- Reports remain basic and synchronous in V1.
